# LLM Audit: Artifact Retrieval-Profile Coverage

## Audit Question

Is `scripts/02.rag-rulebook/report-artifact-retrieval-profile-coverage/script.sh`
strong enough to guide the artifact metadata retrieval-profile migration?

## Method

The audit compared the deterministic coverage report with selected artifact
bodies and headers. The sample covered:

- `strong` records for scripts, workflows, rules, schemas, guides, and selector
  evaluation fixtures
- `partial` records for capability READMEs, script-domain READMEs, templates,
  configs, source material, agent contracts, and rubrics
- the single `excluded` deprecated artifact

This was a calibration review. LLM findings are not a runtime gate. Findings
that were stable enough became deterministic reporter rules.

## Verdict

The reporter is strong enough for migration triage.

It correctly separates header identity health from retrieval-profile derivation
strength. The audit did not find sampled artifacts that need immediate header
repair. The remaining weakness is mostly deterministic generator coverage for
artifact shapes whose bodies already follow stable conventions.

The reporter should not be treated as semantic proof that every generated
profile is complete. It is a migration compass: it tells us where deterministic
profile derivation is thin.

## Findings

### 1. Header Repair Is Not The Current Bottleneck

Sampled partial records had valid identity metadata and meaningful `used_by`
links. Their weakness was the reporter's limited inference, not missing v2
headers.

Decision: keep the next migration slice focused on generator rules before
schema or header backfill.

### 2. Strong Records Were Generally Correct

Sampled strong records matched the artifact bodies:

- service/query scripts adapt or serve context-packet behavior
- workflows define governed process contracts
- rules define structured rulebook obligations
- schemas define machine-readable conformance contracts
- selector fixtures validate retrieval behavior

Decision: keep `strong` as a useful triage label, but inspect profile fields
when a strong record is used as proof for a new retrieval behavior.

### 3. `rulebook` Was A Noisy Script Action Alias

The reporter treated `rulebook` as a `rule` action term, which produced noisy
roles such as `script.rule` for RAG rulebook service scripts.

Deterministic change applied:

- removed `rulebook` from the `rule` action alias set

### 4. Service Artifacts Needed A Deterministic Service Role

The local HTTP service files were classified strongly because they had effects
and references, but their generated fields were too thin for what their bodies
actually do.

Deterministic change applied:

- added a `serve` action for service/server/http terms
- derived service runtime production and request/runtime-boundary validation

### 5. Templates, Configs, And Rubrics Were Under-Modeled

Templates, configs, and rubrics are not merely identity records. Their bodies
are deterministic contracts:

- templates define reusable instance shape
- configs define machine-readable configuration expectations
- rubrics define scoring expectations

Deterministic change applied:

- templates now produce reusable template contracts and validate template
  instance expectations
- configs now produce machine-readable configuration contracts and validate
  configuration expectations
- rubrics now produce review scoring rubrics and validate scoring expectations

## Coverage Impact

Before LLM-calibrated deterministic updates:

- artifacts: `692`
- `strong`: `545`
- `partial`: `146`
- `weak`: `0`
- `excluded`: `1`

After deterministic updates:

- artifacts: `692`
- `strong`: `579`
- `partial`: `112`
- `weak`: `0`
- `excluded`: `1`

No authored headers were changed for this reduction.

## Remaining Generator Queue

The remaining `partial` artifacts are mostly:

- capability READMEs
- script-domain READMEs
- layer/root READMEs
- agent role contracts
- recognition sources
- ADRs
- source-material records that do not yet have a richer deterministic role

These should be handled by explicit generator rules or fixtures only when a
retrieval question proves the extra profile detail is useful.

## Recommendation

Use this reporter as a read-only migration guide and local development check.
Run LLM calibration only when changing the reporter's derivation model or when
coverage results conflict with retrieval evaluation evidence.

Do not add LLM review to the normal gate until there is a separate governed
review workflow, stable prompt, sample policy, and acceptance rubric.
